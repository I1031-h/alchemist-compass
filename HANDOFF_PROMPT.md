## 📝 作業フロー

### 1. 状況確認（必須）
```
以下の順序でドキュメントを確認してください：

1. @PROGRESS.md
   - 完了したタスク
   - 進行中のタスク
   - 未解決の問題
   - 次のステップ
   - 現在のブランチとPR状況

2. @alchemist-compass-docs.md
   - プロジェクトの全体像とビジョン
   - 技術アーキテクチャ
   - 設計思想と決定事項
   - ユーザーコンテキスト
   - 実装状況とロードマップ

3. @アルケミ修正案.md
   - 修正要件の詳細
   - 優先順位
   - 作業ルール（ブランチ管理など）
```

### 2. ブランチ作成（重要：必ずmainから派生）
```bash
# 現在のブランチを確認
git branch

# mainブランチに切り替え
git checkout main

# 最新の状態を取得
git pull origin main

# 新しいブランチを作成（命名規則: fix/feature-name または feat/feature-name）
git checkout -b fix/your-feature-name

# 作業ブランチ名を明示（チャットで報告）
# 例: "現在、fix/max-tokens-prompt-optimization ブランチで作業中です"
```

### 3. 作業とコミット
```bash
# 変更をコミット
git add -A
git commit -m "fix: your commit message"

# プッシュ
git push -u origin fix/your-feature-name
```

### 4. プルリクエスト作成
- GitHubでPRを作成
- 説明に変更内容を記載
- mainブランチにマージ

## ⚠️ 重要な注意事項（アルケミ修正案.mdより）

1. **ブランチ管理（必須）**
   - **必ずmainブランチから新しいブランチを派生**
   - 作業ブランチ名をチャットで明示する
   - 例: "現在、fix/max-tokens-prompt-optimization ブランチで作業中です"
   - 理由: 開発途中で長さ制限に達し、未完成のままデプロイしてしまうことを防ぐため

2. **プルリクエスト（必須）**
   - **必ずPRを作成してからmainブランチにマージ**
   - PRの説明に変更内容を詳細に記載
   - テスト結果も記載

3. **コミットメッセージ**
   - 形式: `fix:`, `feat:`, `refactor:`, `docs:` など
   - 簡潔で明確に
   - 変更の理由も含める

4. **ドキュメント参照**
   - 作業前に必ず3つのドキュメントを確認
   - `PROGRESS.md` - 現在の状況
   - `alchemist-compass-docs.md` - プロジェクト全体像
   - `アルケミ修正案.md` - 修正要件

## 💡 効率的な作業のコツ

1. **3つのドキュメントを順番に読む**
   - `PROGRESS.md` → 現在の状況を把握
   - `alchemist-compass-docs.md` → プロジェクト全体像を理解
   - `アルケミ修正案.md` → 修正要件を確認

2. **作業ブランチ名を明示**
   - チャットで「現在、○○ブランチで作業中です」と報告
   - 複数のブランチがある場合、どれで作業しているか明確に

3. **小さな単位でコミット**
   - 進捗を記録
   - ロールバックしやすくする

4. **PRで進捗を共有**
   - 途中でもPRを作成可能（Draft PRも可）
   - 未完成でも進捗を可視化

5. **エラーは詳細に記録**
   - 再現手順を含める
   - エラーメッセージ全文を記録
   - 環境情報も含める

6. **設計思想を尊重**
   - `alchemist-compass-docs.md`の設計思想に従う
   - 既存パターンに合わせる
   - ユーザーコンテキストを考慮
